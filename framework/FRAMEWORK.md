## エージェント構成

### 規模別エージェント数

| 規模 | 期間 | Lead | Workers | Reviewers | Memory | 合計 |
|------|------|------|---------|-----------|--------|------|
| **Solo Sprint** | 1日 | 1 | 1 | 0 | 0 | **2** |
| **Mini Project** | 1週間 | 1 | 3 | 1 | 0 | **5** |
| **Standard Project** | 1ヶ月 | 1 | 5 | 2 | 1 | **9** |
| **Large Project** | 3ヶ月+ | 1 | 8 | 3 | 2 | **14** |

### 役割定義

#### Lead Agent（企画・分解・判断）
- **担当**: 何をすべきか / なぜすべきか / 優先度
- **入力**: ゴール・制約・過去のメモリ
- **出力**: Plans.md / Sprint Contract / タスクリスト
- **ルール**: 実装には手を出さない。分解と判断に集中

#### Worker Agent（実装・実行）
- **担当**: どうやるか / 実行
- **入力**: タスク定義 / Sprint Contract / ファイルリスト
- **出力**: コード / テスト / ドキュメント
- **ルール**: 担当ファイル外は変更しない。完了後に self_review を返す

#### Reviewer Agent（検証・品質）
- **担当**: 正しいか / 品質基準を満たすか
- **入力**: Worker の成果物 / テスト結果
- **出力**: APPROVE / REQUEST_CHANGES + 具体的指摘
- **ルール**: 感情なし。基準に対して事実のみ報告

#### Memory Agent（学習・蓄積）
- **担当**: 何を学んだか / 次回への改善
- **入力**: 完了タスク / バグ / 成功パターン
- **出力**: `.claude/agent-memory/` へのメモリファイル
- **ルール**: 再現性のある知識のみ記録。一時情報は書かない

## ユースケース別実行プロセス

### UC-05: テストスイート自動生成

```
THINK
  Lead: 対象モジュールのカバレッジ計測 → ギャップ特定
  → "このファイルのテストを書く。優先: 境界値・エラーケース"

ACT
  Worker A: ハッピーパステスト生成
  Worker B: エラーケース・境界値テスト生成 (並列)
  → pytest で自動実行 → カバレッジレポート

LEARN
  Memory: "このモジュールの典型的エラーパターン" を記録
```

### UC-06: API コントラクトファースト開発

```
THINK
  Lead: OpenAPI spec をレビュー → エンドポイント一覧
  → タスクを endpoint 単位に分解（3〜5個並列）

ACT
  Worker A: /users エンドポイント実装
  Worker B: /orders エンドポイント実装  } 並列
  Worker C: 共通バリデーション実装      }
  Reviewer: コントラクト準拠チェック

LEARN
  Memory: "このAPIの認証パターン・エラーフォーマット"
```

### UC-12: インシデント対応自動化

```
THINK
  Lead: ログを構造化 → 仮説3つ生成 → 優先度付け

ACT
  Worker A: ログ解析 → 根本原因特定
  Worker B: 再現テスト作成 (並列)
  Worker C: 修正実装
  → 修正後に自動検証

LEARN
  Memory: "このエラーパターンの対処法" feedback として記録
  → 次回は THINK フェーズで即座に参照
```

## 実装ガイド

### ディレクトリ構造

```
your_project/
├── .claude/
│   ├── agent-memory/          ← Memory Agent の出力先
│   │   ├── MEMORY.md          ← インデックス
│   │   ├── feedback_*.md      ← バグパターン・修正知識
│   │   └── project_*.md       ← プロジェクト状況
│   ├── state/
│   │   └── contracts/         ← Sprint Contract (Lead → Worker)
│   └── rules/
│       ├── test-quality.md
│       └── implementation-quality.md
├── Plans.md                   ← Lead Agent が管理するタスクリスト
└── framework/
    ├── FRAMEWORK.md           ← 本ファイル
    ├── agents/roles/          ← 各 Agent の CLAUDE.md
    ├── scripts/               ← 自動化スクリプト
    └── templates/             ← Sprint Contract テンプレート
```

### 1 タスクの標準フロー

```bash
# 1. Lead: タスク定義 (Plans.md に追記)
# 2. Lead: Sprint Contract を生成
bash framework/scripts/create-contract.sh <task-id> <description>

# 3. Worker: 実装 (並列実行可)
bash framework/scripts/run-worker.sh <task-id>

# 4. Reviewer: 検証
bash framework/scripts/run-reviewer.sh <task-id>

# 5. Memory Agent: 学習記録
bash framework/scripts/record-memory.sh <task-id> <outcome>
```

## 個人開発者がチーム規模を出すための 3 原則

### 原則 1: 直列を並列に変える
- 調査・実装・テストを同時進行させる
- 1つの大タスク → 3〜5の小タスク → 並列 Worker

### 原則 2: 失敗を資産に変える
- バグを潰したら必ず Memory Agent が記録
- 次の同類タスクは「過去の失敗付き」で開始
- 学習の複利が効く

### 原則 3: 人間は判断のみ、実行はエージェント
- APPROVE / REJECT の判断は人間
- 実装・テスト・ドキュメントはエージェント
- 人間がボトルネックにならない設計
